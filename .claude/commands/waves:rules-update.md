# Command: /waves:rules-update

You are executing the waves rules update command. Follow these instructions exactly.

## Your Role

You are the main orchestrator for rules updates. You detect code changes and compare them against existing rules to propose additions, modifications, and deprecations.

## Step -1: Prerequisites Check (CRITICAL)

1. Check if `ai_files/user_pref.json` exists.
   - IF NOT EXISTS → Display: "⚠️ Run /waves:project-init first." → EXIT COMMAND

2. Read `ai_files/user_pref.json`:
   - Extract `user_profile.preferred_language` → Use for all interactions
   - Extract `project_context.project_type` → Determines flow

3. Check rules file exists:
   - Software → `ai_files/project_rules.json`
   - General → `ai_files/project_standards.json`
   - IF NOT EXISTS → Display: "⚠️ No rules found. Run /waves:rules-create first." → EXIT COMMAND

**From this point, conduct ALL interactions in the user's preferred language.**

## Step 0: Command Explanation

Display:
```
📘 Command: /waves:rules-update

I'll detect code changes and compare them against your existing
rules to find new patterns, necessary modifications, or
deprecated practices.

📁 Rules file: [rules_path]
📊 Current rules: [count by section]

Continue? (Yes/No)
```

IF No → Exit.

## Step 1: Fork by Project Type

- IF `project_type === "software"` → Go to **Flow A or B** (depending on git availability)
- IF `project_type === "general"` → Go to **Flow C**

---

## Flow A: Software (Git-based)

### Step A1: Detect Changed Files

1. Check `.git` exists. If not → go to Flow B.
2. Read existing rules and find the oldest `updated_at` or `created_at` timestamp.
3. Use logic from **git-history-analyzer** (see `subagents/17-git-history-analyzer.md`) to get file changes since that date.
4. Use logic from **autogen-detector** (see `subagents/18-autogen-detector.md`) to filter autogenerated files.
5. Filter to code files only (exclude docs, configs unless they affect rules).

### Step A2: Map Changes to Layers

Using the manifest's `architecture_patterns_by_layer`:
- Map each changed file to its architectural layer
- Only re-analyze layers that have changed files
- Display which layers are affected

### Step A3: Re-analyze Affected Layers (scoped)

For each affected layer, invoke IN PARALLEL (same as rules-create but scoped to changed files):
- **pattern-extractor** (see `subagents/22-pattern-extractor.md`) — scoped to changed files
- **convention-detector** (see `subagents/23-convention-detector.md`) — scoped to changed files
- **antipattern-detector** (see `subagents/24-antipattern-detector.md`) — scoped to changed files

### Step A4: Compare with Existing Rules

Use the **rule-comparator** subagent (read from `subagents/27-rule-comparator.md`).

Pass: `existing_rules`, `detected_patterns`, `detected_conventions`, `detected_antipatterns`, `changed_files`.

### Step A5: Present Proposals

Display:
```
📋 Rules update proposals:

═══════════════════════════════════════
➕ NEW RULES ([count])
═══════════════════════════════════════
[For each: section, proposed description, evidence]

═══════════════════════════════════════
✏️ MODIFICATIONS ([count])
═══════════════════════════════════════
[For each: rule ID, current vs proposed, evidence]

═══════════════════════════════════════
⚠️ DEPRECATIONS ([count])
═══════════════════════════════════════
[For each: rule ID, current rule, evidence for removal]

═══════════════════════════════════════
🔴 NEW ANTIPATTERNS ([count])
═══════════════════════════════════════
[For each: location, problem, suggestion]

Options:
1. Apply all proposals
2. Review each one
3. Cancel
```

### Step A6: Apply Updates

**SCOPE PROTECTION:** Rules with `"scope": "ecosystem"` must NOT be modified, deleted, or rephrased. If the analysis detected a change that would affect an ecosystem rule, flag it to the user: "This change affects ecosystem rule #[id]. Ecosystem rules can only be modified from the organization's source of truth, not from individual projects." Skip it unless the user explicitly overrides.

For approved changes:
1. Add new rules with next available IDs. Ask scope for each new rule: "ecosystem" or "local".
2. Update modified rules (set `updated_at`). Only local-scope rules.
3. Remove deprecated rules. Only local-scope rules.
4. Maintain ordering: ecosystem rules first, then local, within each section.
5. Validate against schema.

### Step A7: Success Message

```
✅ Rules updated!

📁 File: ai_files/project_rules.json

📊 Changes applied:
  • New rules: [count]
  • Modified: [count]
  • Deprecated: [count]

⚠️ Antipatterns found: [count]
(Review for code quality improvements)
```

## Flow B: Software (Timestamp-based)

Same as Flow A but uses file modification dates instead of git history.
Use **timestamp-analyzer** (see `subagents/20-timestamp-analyzer.md`) instead of git-history-analyzer.

## Flow C: General Projects (Conversational)

### Step C1: Show Current Standards

```
📋 Your current standards:

[List all current standards by category]
```

### Step C2: Ask What to Update

```
What would you like to update?

1. Add new standards
2. Modify existing standards
3. Remove obsolete standards
```

### Step C3: User Describes Changes

Accept free-form input and structure it using the **standards-structurer** subagent (see `subagents/26-standards-structurer.md`).

### Step C4: Apply and Confirm

Update the standards file. Show summary and confirm.
